## Implementation Plan: `data_integrity_check` MCP Tool

### Overview
New file + 2 modified files. Handler checks 10 categories of integrity issues, optionally auto-repairs. Follows existing conventions exactly.

### Files to Create

#### 1. `src/integrity.ts` (new — ~250 lines)

Exports a single async handler: `handleDataIntegrityCheck(args)`.

**Checks performed (in order):**

| # | Check | SQL/Logic | Repair action |
|---|-------|-----------|--------------|
| 1 | **Invalid status** | `SELECT * FROM tasks WHERE status NOT IN (valid enum values)` | Set status → `'pending'` |
| 2 | **Orphan parent_id** | `SELECT * FROM tasks WHERE parent_id IS NOT NULL AND parent_id NOT IN (SELECT id FROM tasks)`, including `id == parent_id` | Set `parent_id = NULL` |
| 3 | **Broken depends_on** | For each task, `JSON.parse(depends_on)`, filter IDs not in tasks table, plus self-refs `depId == task.id` | Remove invalid + self-ref IDs from array, deduplicate |
| 4 | **Duplicate depends_on** | For each task, `JSON.parse(depends_on)` has duplicate entries | Deduplicate array |
| 5 | **Retry count exceeded** | `SELECT * FROM tasks WHERE retry_count > max_retries` | Clamp `retry_count = max_retries` |
| 6 | **Orphan snapshots** | `SELECT * FROM snapshots WHERE task_id NOT IN (SELECT id FROM tasks)` | `DELETE FROM snapshots WHERE ...` |
| 7 | **Orphan audit_log** | `SELECT * FROM audit_log WHERE task_id NOT IN (SELECT id FROM tasks)` | `DELETE FROM audit_log WHERE ...` |
| 8 | **Broken tool_name refs** | `SELECT * FROM tasks WHERE tool_name IS NOT NULL AND tool_name NOT IN (SELECT name FROM tools)` | `SET tool_name = NULL` |
| 9 | **Circular parent refs (direct)** | `WITH RECURSIVE` cycle detection on parent_id chain | Report only (too dangerous to auto-break) |
| 10 | **Circular dependency (depends_on)** | Build adjacency from depends_on, DFS detect cycles | Report only (too complex to auto-fix) |

**For each issue found, push to `issues[]` with shape:**
```typescript

```

At the end, compute: `healthy = issues.length === 0`.

If `repair === true`, wrap all repair mutations in `db.transaction(...)` for atomicity, then re-run the checks to generate a final report reflecting the repaired state.

### Edge Cases Handled

- **Empty DB**: zero issues found, `healthy: true`
- **Malformed `depends_on` JSON**: caught by `JSON.parse` try-catch, reported as its own issue type `'malformed_depends_on'`, repair restores to `'[]'`
- **Self-referencing `parent_id`**: caught by orphan check (`id == parent_id`), repair sets to NULL
- **Circular dependency chains of arbitrary length**: DFS with visited set, cycle detection via ancestor tracking
- **Circular parent chains**: recursive CTE detects if any task is its own ancestor in parent_id chain
- **Concurrent mutation safety**: repairs wrapped in `db.transaction()` (SQLite serializes writes)
- **Numeric overflow**: `retry_count` and `max_retries` are bounded integers by schema; clamp uses `Math.min(retry_count, max_retries)` which is safe in JS for SQLite INTEGER range
- **depends_on with empty array `[]`**: trivially passes checks
- **audit_log entries with no matching task** (no FK constraint): repaired by deletion
- **tool_name referencing tool that was later deleted**: repaired by NULLing it
- **Multiple issues on the same task**: each issue type reported separately; repairs applied in sequence within the transaction
