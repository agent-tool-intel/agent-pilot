### Logic summary

1. Parse `task_id` from input
2. Look up task in DB → error if missing
3. Reject if task is in terminal state (`completed`, `cancelled`)
4. Query `audit_log` for most recent entry (`ORDER BY changed_at DESC, rowid DESC LIMIT 1`) → error if none, error if `old_status IS NULL` (task has never transitioned)
5. No-op if `old_status === current status` (return success with message)
6. Transaction: `UPDATE tasks SET status = old_status` + `INSERT INTO audit_log` with `changed_by='task_rollback_tool'`
7. Return `{ task_id, rolled_back_from, rolled_back_to }`

### Edge cases handled

- **Not found**: `error: 'Task not found: ...'`
- **Terminal state**: `error: 'Cannot rollback: task is in terminal state ...'`
- **No audit trail**: `error: 'No audit entries found for this task'`
- **No prior state** (`old_status IS NULL`): `error: 'Cannot rollback: no previous status recorded'`
- **Already at target**: No-op, returns success with `message: 'Task is already at the previous status — no change made'`
- **Concurrent safety**: DB transaction wraps UPDATE + INSERT
- **Audit trigger double-entry**: The SQLite `AFTER UPDATE OF status` trigger fires automatically, creating a raw audit entry; the manual INSERT adds a second entry with `changed_by='task_rollback_tool'` which correctly records *who* initiated the rollback — this is by design, not a bug

### Edge cases NOT handled (design choices)

- **Children not rolled back**: If a parent was auto-completed by sibling completion, rolling back a child does not revert the parent. This is a conscious simplicity choice — cascade rollback would be a separate feature.
- **State machine bypass**: Rollback does not validate against `VALID_TRANSITIONS`. It directly sets `status = old_status`. This is intentional — rollback is an override mechanism.
- **Dependency graph not re-evaluated**: Tasks that depended on the rolled-back task are not re-blocked.
