**5. Dual-Review Workflow**
Agent A (executor): task_next(mode=executor) → execute → task_update(status=pending_review)
Agent B (reviewer): task_next(mode=reviewer) → review
  → task_update(status=approved) OR task_update(status=needs_revision, review_comment=...)
Auto-cascade: when all subtasks approved/completed, parent auto-completes.

**6. Model Router**
- What it is: Zero-LLM keyword-based rule engine → 8 categories
- Plan A (International): Claude Opus 4.7, Sonnet 4.6, GPT-5.4, Gemini, DALL-E 4
- Plan B (China): DeepSeek V4-Pro, V4-Flash, GLM-5.1, GLM-4.6V, CogView4, Kimi K2.5
- Plan C (GLM Lazy): All GLM series — GLM-5.1, GLM-5, GLM-4-Flash, GLM-4.6V
- How to switch: `model_config action=switch plan=B`
- Cost savings: Plan A uses expensive flagship for reasoning only, cheap for reading (Gemini $0.40 → Claude $15 = 97% savings). Plan B costs 1-2 orders of magnitude less than Plan A (DeepSeek ¥2 vs OpenAI $15). Dynamic routing reduces blended cost by ~60% vs always-strongest.

**7. Architecture — The Infrastructure Moat**
- SQLite: persistent task state survives model restarts, full audit trail with triggers, WAL mode for concurrent access, recursive CTE for tree operations
- State machine: enforced valid transitions, parent auto-cascade, dual-review states
- FTS5: full-text search over tool registry, relevance scoring — model upgrades cannot create this
- Rule engine (model_classify): 8-category keyword classifier with zero LLM dependency — models cannot replace deterministic matching for routing
